# Supabase AI Guide

This is the Supabase-specific AI and maintainer guide for this repository.

Use it together with `docs/STARTER_KIT_AI_RULES.md`. The generic rules define how starter-kit repositories should behave; this file defines the concrete source-of-truth and Supabase-specific constraints for this repo.

## Repository identity

- This is a Supabase-only self-hosted starter.
- Do not convert it into a multi-app monorepo.
- Prefer upstream Supabase-compatible images and configs.
- Keep local Compose, Dokploy and generated docs synchronized.

## Current source of truth

| Area | Source |
|---|---|
| Canonical stack | `compose.yaml` |
| Production override | `compose.prod.yaml` |
| Env inventory | `.env.example` |
| Runtime bind files | `files/` |
| Service/image inventory | `manifest.yaml` |
| Dokploy source config | `dokploy/config.toml` |
| Dokploy generated artifact | `dokploy/template.json` |
| Runtime tests | `tests/runtime/` |
| Render/validation scripts | `scripts/` |
| README generated summary | generated from `manifest.yaml` by `scripts/render-readme.sh` |
| Documentation policy | `docs/DOCUMENTATION_STRUCTURE.md` |

Generated artifacts:

- `dokploy/template.json` is generated by `scripts/render-dokploy.sh`.
- Do not edit generated artifacts directly unless the renderer changes in the same commit.

## Required workflow for AI agents

1. Run or inspect `git status --short`.
2. Determine whether the request affects Compose, env, files, platform artifacts, tests, scripts or docs.
3. Update all synchronized artifacts, not just the file that looks directly related.
4. If `compose.yaml` or `files/` changed, run:

   ```sh
   ./scripts/render-dokploy.sh
   ```

5. If `manifest.yaml` or README summary data changed, run:

   ```sh
   ./scripts/render-readme.sh
   ```

6. Always run:

   ```sh
   ./scripts/validate.sh
   ```

7. If a live deployment is available and the user authorizes runtime testing, run:

   ```sh
   npm --prefix tests/runtime test
   ```

8. Report what changed, what was validated and what still needs fresh deploy/runtime testing.

## Supabase-specific constraints

- Kong is the only public ingress by default.
- Postgres, MinIO, Logflare, pg-meta and internal service ports must not be publicly exposed by default.
- Studio is exposed through Kong and protected by basic auth.
- Modern opaque keys are supported at Kong and mapped internally to legacy Supabase JWT keys.
- Do not add gateway `key-auth` to Storage routes if it breaks S3/SigV4 compatibility.
- GraphQL is enabled through `pg_graphql` and `graphql_public.graphql`.
- `pg_cron` is enabled with admin-focused permissions.
- SMTP and phone signup are disabled by default.
- Public signup defaults must remain safe until real SMTP/SMS configuration exists.
- Admin and migration routes must require service/admin credentials.

## Env source-of-truth

`.env.example` is the full non-secret inventory.

Real env generation:

```sh
SUPABASE_PUBLIC_URL=https://example.com ./scripts/generate-env.sh
```

The generator:

- refuses to overwrite `.env` unless `FORCE=1`;
- creates strong secret values;
- creates anon and service-role JWTs signed by the generated JWT secret;
- keeps required alias values identical.

Important aliases:

| Canonical value | Required aliases |
|---|---|
| Postgres password | `POSTGRES_PASSWORD`, `SERVICE_PASSWORD_POSTGRES` |
| JWT secret | `JWT_SECRET`, `SERVICE_PASSWORD_JWT` |
| Anon key | `ANON_KEY`, `SERVICE_SUPABASEANON_KEY` |
| Service role key | `SERVICE_ROLE_KEY`, `SERVICE_SUPABASESERVICE_KEY` |
| Dashboard user | `DASHBOARD_USERNAME`, `SERVICE_USER_ADMIN` |
| Dashboard password | `DASHBOARD_PASSWORD`, `SERVICE_PASSWORD_ADMIN` |
| MinIO user | `MINIO_ROOT_USER`, `SERVICE_USER_MINIO` |
| MinIO password | `MINIO_ROOT_PASSWORD`, `SERVICE_PASSWORD_MINIO` |

## Static validation contract

`scripts/validate.sh` must validate:

- root Compose syntax;
- production override syntax;
- Dokploy template JSON;
- embedded Dokploy Compose syntax;
- required bind-mounted files.

CI must fail if generated Dokploy or README artifacts drift from their renderers.

## Runtime test contract

Runtime tests live in `tests/runtime/` and use Vitest with TypeScript.

Runtime test usage is documented in `tests/runtime/README.md`.

The suite should cover:

- Studio basic auth;
- Auth health;
- signup disabled defaults;
- admin user create/delete;
- REST OpenAPI and CRUD;
- GraphQL introspection;
- Storage bucket/object CRUD;
- signed URL path;
- image transformation path;
- Edge Function hello route;
- Realtime WebSocket/channel path;
- pg-meta admin ACL;
- `pg_cron` schedule/list/unschedule.

Runtime tests are allowed to create temporary resources and must clean them up.

Static test-package validation:

```sh
npm --prefix tests/runtime run typecheck
```

## Platform notes

### Dokploy

Dokploy has a real generated import artifact:

```text
dokploy/template.json
```

Render it with:

```sh
make render-dokploy
```

The renderer:

- reads `compose.yaml`;
- reads `dokploy/config.toml`;
- rewrites `./files/...` to `../files/...`;
- embeds `files/` content into `[[config.mounts]]`;
- runs static validation.

Operational Dokploy usage is documented in `dokploy/README.md`.

### Coolify

This repository does not keep a separate Coolify artifact.

Use root `compose.yaml` as the manual/user-defined Compose base. This is not a Coolify one-click catalog template. Operators must ensure `files/` exists relative to the deployment bundle or adapt bind paths on the target platform.

## Branch and versioning policy

This repository uses simple project-specific versioning.

Branches:

- `main`: canonical stable development branch.
- `feature/<short-scope>`: scoped feature work.
- `fix/<short-scope>`: scoped bug/security/config fixes.
- `docs/<short-scope>`: documentation-only changes.
- `experiment/<short-scope>`: disposable trials that must not be treated as release candidates.

Tags:

- Use immutable tags for known-good states.
- Recommended tag shape: `supabase-starter-vX.Y.Z`.
- If a tag is tied to a major upstream stack refresh, use: `supabase-starter-vX.Y.Z-upstream-YYYY.MM.DD`.

Version source:

- `manifest.yaml` owns `starter_version` and `last_updated`.
- `manifest.yaml` owns service image tags and public access metadata.
- `README.md` displays generated version/service summary from `manifest.yaml`.

Version bump rules:

- Patch bump: docs, tests, script fixes, non-breaking config cleanup.
- Minor bump: new supported feature, new platform integration, materially improved production behavior.
- Major bump: breaking env/volume/network/template behavior or major Supabase stack migration.

Release checklist:

1. Update image tags and metadata in `manifest.yaml`.
2. Run `make render-readme`.
3. Run `make render-dokploy` if Compose or files changed.
4. Run `make validate`.
5. Run runtime smoke tests when a deployment is available.
6. Create an immutable git tag only after static validation and, for production-ready states, runtime/fresh deploy testing.

## Backup/restore contract

Stateful services currently requiring backup coverage:

- Postgres named volumes;
- MinIO named volume.

Scripts:

- `scripts/backup-postgres.sh`
- `scripts/restore-postgres.sh`
- `scripts/backup-minio.sh`
- `scripts/restore-minio.sh`

Restore scripts must remain guarded by `CONFIRM_RESTORE=1`.

Script usage and safety behavior are documented in `scripts/README.md`.

| Dokploy behavior | `dokploy/README.md`, render script, generated template |
| Runtime test behavior | `tests/runtime/README.md`, package scripts, CI typecheck |

## Do not

- Do not add unused folders.
- Do not commit real secrets.
- Do not add Dockerfiles unless official images are insufficient.
- Do not rely on hidden Compose fallbacks for production-critical env.
- Do not expose internal services publicly.
- Do not weaken admin ACLs for convenience.
- Do not mark the stack production-ready after static validation only; fresh deploy and runtime tests are separate.
